# 验收用例设计

当任务会被用户按产品视角验收，而不是只按代码视角验收时，使用这个技能。

目标不是写“测试会覆盖”，而是把“如何证明它真的对”变成逐条可执行的用例矩阵。

## 适用场景

- Web / App / 桌面 / CLI 产品
- 页面、组件、表单、聊天、工作流
- 需要交互、视觉、状态、权限、异常分支的功能
- 用户可能从真实使用中发现大量 bug 的任务

## 核心原则

- 验收标准必须落到具体场景，不停留在抽象条款
- 每条用例都要能回答“用户怎么做”“会看到什么”“最终应该发生什么”
- 特殊案例要前置写出来，不留给用户在验收时随机发现
- 视觉与交互结果也是验收结果，不只是“功能成功”

## 输出格式

至少形成一个验收用例矩阵，字段建议为：

- 用例编号
- 场景 / 用户旅程
- 前置条件
- 用户动作
- 预期视觉结果
- 预期交互状态
- 预期功能结果
- 特殊案例 / 风险点
- 验证方式

## 设计步骤

1. 先列主黄金路径
2. 再列最脆弱生命周期路径
3. 再补异常分支、空态、慢响应、恢复、权限、边界输入
4. 把这些用例映射到 requirements / PRD / tech spec / plan / tracker
5. 每个实现任务至少绑定一部分用例编号

## 必须覆盖的维度

### 1. 用户旅程

- 新建
- 编辑 / 继续
- 删除 / 退出 / 完成
- 恢复 / 重进

### 2. 视觉结果

- 页面展示什么
- 信息层级是否合理
- 中文阅读是否顺畅
- 移动端 / 桌面端布局是否正常

### 3. 交互状态

- loading
- disabled
- empty
- error
- partial
- success

### 4. 功能结果

- 数据是否真的落账
- 状态是否真的更新
- 权限是否真的生效
- 外部系统是否真的完成

### 5. 特殊案例

- 边界输入
- 慢响应
- 旧数据 / 大量历史
- 断线 / 恢复
- 重复点击
- 首次使用与已有对象继续使用

## 什么时候说明验收设计不足

- 只有“应该支持 X”这种抽象句子
- 没写视觉 / 交互预期
- 没写异常分支
- 没写特殊案例
- 修 bug 后没有把用户报告路径补进回归矩阵

## 交付要求

最终结果必须能直接被后续技能使用，而不是一篇空泛说明。

至少让后续实现者可以据此回答：

- 先做哪几条用例
- 每条用例由什么验证手段覆盖
- 用户下一次怎么验收
