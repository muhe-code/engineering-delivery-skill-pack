# 做到真正完成

当用户是在要求你交付结果，而不是闲聊时，使用这个技能。

核心目的很简单：如果任务有明确终点，就一直做到终点。不要停在“已经有不错进展”“第一版出来了”“主体完成了”“如果你愿意我还可以继续”这种半截状态上，只要剩余工作本来就在请求范围内，就应继续做完。

开始前先阅读 `references/closure-rules.md`。

## 何时使用

适用于：

- 工程与实现类工作
- 需要产出推荐或结论的调研
- 需要形成可直接使用成品的写作
- 需要走到已验证解决方案或明确阻塞报告的调试
- 需要形成可执行计划而不是停留在脑暴残留的规划
- 有明确目标状态的运维/操作任务
- “部分完成”不等于“完成”的多步骤任务

不适用于：

- 轻松闲聊
- 用户明确只想做开放式脑暴
- 用户明确要求停在某个中间步骤

## 基本规则

在开始前，先在内部明确：

1. 用户最终希望拿到什么
2. 对当前任务来说，“完成”具体意味着什么
3. 需要什么证据或质量检查才能支撑“已完成”的说法
4. 哪些后续步骤属于范围内，哪些只是额外加料
5. 交付物的语言、格式、受众是否已经明确

然后一直推进，直到范围内工作被真正关闭。

如果用户明确表达过“你不懂我”“你不通人意”“你只是在机械干活”，还要额外明确：

6. 用户嘴上给出的任务，和他真正想达成的结果是否一致
7. 用户最怕发生什么，即哪些结果在技术上可运行、但在用户心里仍然算失败
8. 用户此刻真正需要的是功能、解释、掌控感、确定性、风险边界，还是被理解
9. 你现在的回答或实现，是否只是解决字面动作，而没有解决他的真实焦虑
10. 最终交付时，要拿什么证据证明“这次不是机械执行，而是真对上了需求”

如果任务包含用户可见的前端 UI、页面布局、交互体验、视觉审美或界面重构，还要额外明确：

6. 这次交付的前端完成标准是什么
7. 需要哪些浏览器级证据来支撑“界面已完成”的说法
8. 是否必须联动前端设计类 skill，而不是只靠工程总控技能硬做
9. 用户最关键的真实使用路径有哪些，是否已经逐条列出
10. 哪些验证只是 mock / fixture / 本地替身，哪些验证真正覆盖了正式入口或生产路径

如果任务依赖第三方 API、远端服务、支付/交易/通知系统、设备状态、部署进程或任何“外部权威真相源”，还要额外明确：

11. 哪个系统的哪个字段才是最终真相源
12. 当前展示给用户的内容里，哪些是推断值，哪些已完成外部对账
13. 出现不一致时，应该把问题归因为投影延迟、外部拒绝、权限问题、字段映射错误还是文案误报

如果任务需要交付可验收产品、功能或界面，还要额外明确：

14. 用户会按哪些逐条场景进行验收
15. 每条场景期望看到什么视觉结果、交互状态和功能结果
16. 哪些特殊案例必须纳入回归，而不是留给用户去踩

如果任务的主要交付物是一组相互依赖的文档链，例如 requirements / PRD / tech spec / plan / tracker，还要额外明确：

17. 这组文档里哪些内容属于“关键决策”，必须在多份文档中保持一致
18. 哪些实现名词属于硬契约，例如状态名、环境变量、命令名、tool id、job 名称、真相源字段
19. 是否仍存在“单篇文档看起来完整，但多篇文档拼起来互相打架”的风险
20. 是否仍存在会阻碍实现的模糊词，例如“前 N 个”“更高频”“适当”“超预算”“默认”等却没有默认值或参数表

如果任务是 agent、bot、自主系统、后台调度系统或任何“会自己持续运行”的产品，还要额外明确：

21. 当前能力点清单是什么，是否已经逐项映射到验证方式，而不是只跑一套大而化之的测试
22. 用户可见输入/输出类型有哪些，是否已经为每一类建立代表性验收样本
23. 外部系统的负路径目录是什么，例如拒单、权限错误、限流、下架、部分成功、保护失败、投影延迟
24. 启用后的真实观察窗口是什么，是否至少观察过一次定时触发、一条真实外部交互和一条恢复/降级路径
25. 当前是否已经有 harness contract：知识入口、一键启动、运行时 signals、eval gate、kill switch、rollback

如果任务改动很小，但仍可能影响工程内其他功能，还要额外明确：

26. 本次改动影响了哪些能力点、共享路径、配置或约束
27. 哪些功能点必须回归，哪些判断为本轮无需回归，以及理由

如果任务需要独立验证或用户已经对结果表达不信任，还要额外明确：

28. 谁负责 Builder、Verifier、Closer 三个责任面
29. `Verifier` 是否由不同于 `Builder` 的执行体承担；若不能，为什么当前任务还不能声称完成
30. 验证问题的状态流转规则是什么，谁有权关闭问题
31. 哪些未关闭问题会阻断完成声明

## 核心行为

### 1. 先定义闭环

在动手前，先明确最终交付物。

如果任务同时包含“结果交付”和“理解用户真实心意”两层诉求，闭环定义必须同时覆盖两者：

- 字面任务是否完成
- 隐含目标是否被识别并转成了实现 / 文档 / 验收标准
- 是否避免了“功能完成但用户仍觉得你没懂他”的假完成

示例：

- 修 bug -> 行为修复并附带验证证据
- 调研 -> 排名后的答案、依据与建议
- 写作 -> 符合指定格式和语气的成稿
- 规划 -> 覆盖范围、步骤清晰、退出条件明确的计划
- 工程功能 -> 完整产物链、实现与验证

### 1.5 流程轻重与思考深度分离

不是每个任务都需要重型框架，但每个任务都需要完整影响分析。

如果改动很小，可以使用轻量记录替代完整文档链；但至少必须写清：

- 改动点
- 影响模块 / 能力点
- 回归点清单
- 本轮验证方式
- 哪些点不回归，以及理由

### 2. 不要把必须做的事包装成可选项

不要把本来就必须完成的后续动作说成“如果你愿意的话”。

反例：

- “如果你愿意，我也可以把缺失的 prompts 补上。”
- “要不要我下一步再验证一下？”
- “你要我继续把剩余 plan items 做完吗？”

如果剩余步骤是完成请求所必需的，就直接做。

### 3. 区分范围内工作和打磨项

不是所有额外工作都必须自动继续。

应自动继续的情况：

- 满足请求所缺失的必要步骤
- 缺失的验证
- 已承诺部分之间还未打通
- review 发现但尚未解决的问题
- 必要的文档或产物更新
- 用户在验收中发现的问题，其复现、修复、回归验证与文档回写

应暂停征求用户确认的情况：

- 新增独立功能
- 与请求无关的视觉/文字打磨
- 推测性的优化
- 无关重构

### 4. 交付物语言必须正确

当任务产出文档、计划、PRD、说明或报告时：

- 默认跟随用户当前会话语言
- 如果仓库已有明确文档主语言，则跟随仓库约定
- 不能在用户使用中文、仓库也无反向约束时，静默产出英文文档
- 代码标识符可以保留英文，但正文、标题、表格字段、review 结论应符合选定语言

语言不匹配，视为未完成。

### 5. 先有证据，再谈完成

如果你声称某件事已经完成、可用、正确、已安装、已生成或已验证，必须持有与任务类型匹配的证据。

证据形式取决于任务：

- 工程：测试、命令、diff、构建产物
- 调研：来源与综合判断
- 写作：最终成稿与格式符合度
- 运维：命令输出或结果状态
- 规划：覆盖请求范围且可执行的明确计划

如果任务涉及前端 UI，证据还必须包含至少一类浏览器级验证，例如页面截图、无头浏览器 smoke、真实交互回放、可读性/响应式检查；只有接口通过，不足以支撑“界面已完成”。

如果任务包含部署、正式地址、线上服务、反向代理、证书、PWA、手机访问或“现在已经可以用了”这类声明，证据还必须额外覆盖：

- 正式地址或正式进程上的真实验证，而不是只验证本地开发态
- 当前部署的静态资源、脚本或进程确实是新版本，而不是旧缓存/旧进程
- 至少一条与用户真实使用一致的黄金路径

如果任务包含聊天、会话、长轮次、轮询、流式、重连、图片、附件、多步骤状态机或“先创建再继续”的对象生命周期，证据不能只覆盖瞬时成功的 happy path：

- 必须覆盖“新建对象后第一次使用”
- 必须覆盖“已有对象继续使用”
- 必须覆盖至少一条慢响应、处理中、重试或恢复路径
- 必须检查用户可见状态是否真的正确变化，例如输入框是否清空、按钮是否禁用、会话是否保持选中、最终结果是否进入正式历史
- 如果 mock / fixture 省略了真实系统中的合并、投影、缓存、持久化或慢响应语义，视为验证不足

如果任务依赖外部权威系统，证据还必须额外覆盖：

- 至少一条写操作后的真实对账，而不是只看本地日志、缓存或通知文案
- 关键用户可见字段与权威来源一致，例如余额、持仓、状态、成交、权限、交付结果
- 如果存在延迟投影，必须明确“当前是待同步”而不是伪装成“已经成功”

如果任务是 agent、bot、自主系统、后台调度系统或强状态化产品，证据还必须额外覆盖：

- 一份“能力点 -> 证据”覆盖账本，而不是只给出总体测试通过
- 每类用户可见 I/O 至少一条代表性样本，例如成功、拒绝、降级、自修复、未达标、恢复完成
- 至少一条负路径证据，而不是只验证 happy path
- 如果依赖 scheduler / worker / timer / webhook / cron，至少观察一轮真实触发完成，而不是只确认配置存在
- 如果启用了自动执行或后台任务，至少给出一次启用后的观测证据，例如日志、状态、对账或用户可见输出
- 至少一条 operator control path 证据，例如 disable、pause、kill switch、rollback
- 至少一条启动与观测证据，证明这不是“只能人工盯住的黑箱系统”

如果用户明确质疑你“不懂真实意图”，证据还必须额外覆盖：

- requirements / PRD / tracker 中是否已经写明隐含目标、不可接受失败和真实成功标准
- 最终实现 / 结论是否真的改善了这些点，而不是只完成了字面事项
- 至少一条最终汇报内容，能把“你真正要的是 X，不只是 Y”说清楚并自证

如果任务交付的是多份互相依赖的文档，证据还必须额外覆盖：

- 至少一次跨文档一致性审计，而不是只逐篇自审
- 关键决策在 requirements / PRD / tech spec / plan / tracker 中没有互相矛盾
- 状态名、环境变量、命令名、tool id、job 名称、外部真相源字段保持一致
- 全链路中不存在 `TBD / TODO / 待确认 / 未定 / 待补` 之类未闭合占位词
- 所有会阻碍实现的模糊词，要么已被替换成默认值/参数表，要么被明确记录为延期且不阻塞当前阶段

如果任务需要独立验证，证据还必须额外覆盖：

- 至少一轮独立验证结论，而不是只看 Builder 自检
- `Verifier` 与 `Builder` 不是同一执行体
- 验证问题的当前状态账本
- 被修复的问题已通过同路径重验
- 完成声明不包含任何仍处于 `open`、`accepted-for-fix` 或 `fixed-awaiting-reverify` 的阻断项

### 6. 只因真正原因暂停

可以暂停的正当理由：

- 缺少用户必须做出的关键决策
- 下一步会超出当前范围
- 破坏性或高风险动作需要明确批准
- 当前环境阻止任务完成
- 有些信息无法安全推断

“我觉得差不多了”不构成暂停理由。

如果唯一理由是“上下文快满了”，也不能直接停下。此时必须先生成交接包，至少包含：

- 当前目标
- 已完成
- 未完成
- 未关闭 verifier issue
- 高风险未覆盖能力点
- 下一步唯一优先动作
- 关键证据路径
- 当前为何不能宣称完成

## 任务类型路由

当下列场景匹配时，走对应专门流程：

- 端到端工程交付：使用 `spec-to-ship`
- 故障与异常行为：使用 `systematic-debugging`
- 必须先完成设计再实现：使用 `brainstorming`
- 多步骤实现规划：使用 `writing-plans`
- 宣称成功前：使用 `verification-before-completion`
- 任何包含 Web 前端、页面、组件、仪表盘、落地页、CSS 布局、美化、重排、信息密度优化、响应式或交互动效的任务：必须额外使用 `frontend-design`
- 当前端任务的重点是审美质感、布局品味、去除 AI 味、提升界面层次、优化视觉节奏、让长内容更易读时：在 `frontend-design` 之外，再额外使用 `design-taste-frontend`
- 任何需要把用户旅程、视觉效果、交互状态、功能结果、异常分支、特殊案例写成逐条验收用例的任务：额外使用 `acceptance-test-design`
- 任何包含聊天、会话、流式、轮询、附件、长任务、状态机、会话恢复、断线重连、乐观更新、分页加载的任务：额外使用 `stateful-product-validation`
- 任何依赖第三方 API、远端服务、云资源、部署进程、支付/交易/通知系统、设备状态或外部权威真相源的任务：额外使用 `external-system-reconciliation`
- 任何 agent、bot、后台调度、长期运行服务或需要先设计运行环境与反馈环的任务：额外使用 `long-running-app-harness`
- 任何需要独立验收、反向找错、阻止实现者自证完成的任务：额外使用 `independent-verification`

不要只启用 `done-means-done` 就直接开始改前端。前端类任务默认需要总控 skill 与前端 skill 联动。

## 完成时的表达

当范围内仍有工作：

- 说清还剩什么
- 做完它
- 做完后再汇报

当工作已完成：

- 说明完成了什么
- 说明有什么证据支撑
- 说明哪些事项被明确延后（如果有）

除非下一步真的超出范围，否则不要以“如果你愿意我可以继续”结尾。

如果用户已经亲自验收并报出问题，不得再用“之前验证通过”来支撑完成声明。此时必须：

- 先复现用户报告的精确路径
- 修复后用同一路径重新验证
- 把这条路径补进可重复的回归验证
- 只有该用户路径也通过后，才能重新声称完成

如果当前任务启用了独立验证者，还必须满足：

- Builder 不能自己把问题标成 `verified` 或 `closed`
- Builder 修复后，状态只能推进到 `fixed-awaiting-reverify`
- Verifier 完成同路径重验后，才能给出 `verified`
- Verifier 必须是不同于 Builder 的执行体；否则只能报告验证条件缺失，不能收尾
- 只要仍有阻断问题未被独立验证关闭，就不得收尾

如果用户报出的不是单一 bug，而是“你整体没懂我要什么”，也不得只修局部功能。此时必须：

- 先回到完成定义，重写隐含目标与失败标准
- 修正技能 / 文档 / 实现中导致错位的部分
- 用新的验收口径重新验证，而不是继续沿用旧的成功标准

如果任务之前没有逐条验收用例矩阵，而现在已经暴露出用户验收 bug，必须补齐这套矩阵；否则只是继续靠运气补洞。

如果用户在中途把产品形态改了，例如原生改 Web、同步改异步、现货改合约、单实例改多实例，也不得把旧的 done definition 沿用到底。此时必须先重定义“这次完成到底是什么”，再继续推进。

如果任务的核心是文档链建设，也不得在“每份文档单独都像样”时就停下。此时必须额外完成：

- 一次 requirements / PRD / tech spec / plan / tracker 的交叉核对
- 一次面向实现的可开发性检查，确认工程师无需再猜关键默认值、配置契约和接口命名
- 一次机器可检查的占位词清理，例如 grep `TBD / TODO / 待确认`

## 反模式

- 用户要成品，你却停在第一版草稿
- 用户要实现，你却停在分析
- 没验证就宣称成功
- 把未解决问题称作“后续优化”
- 对已经约定范围内的工作再次询问是否继续
- 交付物语言与用户/仓库约定不符
- 让用户在实现阶段继续补回本应前置澄清的关键信息
- 用户明确嫌界面乱、丑、杂或不易读，你却只做功能层修补，没有启用前端设计类 skill
- 只做 API、状态管理或样式微调，就宣称前端体验已经完成
- 只做本地验证、mock 验证或 happy path 验证，就宣称正式环境已经完成
- 用户实际通过手机、浏览器或正式地址使用，你却没有验证正式入口
- 验证只覆盖已有对象，不覆盖“新建后立即使用”“首轮消息”“首次打开详情”这类脆弱路径
- mock fixture 缺少真实环境的关键语义，却仍被当成强验收证据
- 部署脚本失败、进程未替换或静态资源未刷新，却仍宣称“已经上线可用”
- 产品形态已经变化，却还拿旧验收标准当作当前完成标准
- 外部系统的真实状态与本地展示不一致，却仍宣称功能正确
- 没有先写出逐条验收用例，就直接开始堆实现并匆忙交付
- 每篇文档单独看都合理，但状态机、环境变量、tool 名称、默认值在多篇文档之间不一致
- 把“前 N 个”“更高频”“超预算”“默认”等实现级模糊词留到编码阶段临场决定

## 底线

完成不是一种感觉。

完成意味着：范围内的目标已经存在、被适当检查过，并且没有把明显属于本次请求的剩余工作悄悄推回给用户。
