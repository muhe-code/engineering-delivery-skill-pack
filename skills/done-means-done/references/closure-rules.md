# 收口规则

用这些规则把任务真正推到终点。

## 1. 先识别交付物

把用户请求翻译成用户最终应该拿走的结果。

示例：

- “找出最好的选项” -> 有依据的推荐结论
- “修一下这个问题” -> 已验证的修复或明确阻塞报告
- “写这个内容” -> 完成稿
- “规划这个事情” -> 可执行计划，包含范围与检查项
- “安装这个” -> 已安装结果与验证证据

## 2. 找到隐藏的最后一公里

过早停下通常发生在最后一公里：

- 草稿有了，但没打磨到可用
- 代码改了，但没验证
- 结论有了，但没排序或没给建议
- 文件建了，但还没接通
- 回答给了，但没对准用户真正要做的决策

最后一公里通常属于任务本身，不是可选项。

## 3. 问自己：剩余工作是必需还是额外？

必需：

- 原始请求已经隐含要求
- 没它结果就不可用
- 没它就无法正当宣称“完成”
- 当前工作暴露出了必须补上的缺口

额外：

- 新的独立功能
- 推测性的改进
- 用户没有要求的偏好型选择

## 4. 分任务类型检查闭环

### 工程

- 实现完成
- 验证已跑
- 关键验收用例已逐条跑过
- review 问题已解决或明确记录
- 文档/产物已同步更新
- 正式入口或真实使用路径（如果存在）已验证
- 至少一个最脆弱生命周期路径已验证，例如“新建后第一次使用”“首次加载详情”“旧对象继续使用”
- mock / fixture 与真实系统的关键语义差异已被显式检查，而不是默认当作等价
- 如果依赖外部权威系统，关键用户可见字段已完成真实对账，而不是只看本地日志或通知文案

### 调研

- 相关来源已收集
- 结论已综合
- 推荐或结论已明确给出
- 不确定性边界已说清

### 写作

- 符合要求格式
- 语气与受众匹配
- 没有尴尬空洞或占位内容
- 结果可直接使用
- 语言符合用户当前语言或仓库约定

### 规划

- 范围覆盖完整
- 步骤可执行
- 依赖与检查项明确
- 完成条件明确
- 阻塞性未决问题已被回问或显式延后
- 语言符合用户当前语言或仓库约定

### 运维

- 请求动作已执行
- 结果状态已检查
- 对用户的影响已总结

## 5. 一个好的终态测试

问自己：

- 如果我现在停下，用户会不会合理地觉得有一部分本该我完成的工作被推回给了他？

如果答案是会，就继续。

额外问自己：

- 我现在的验证，是否真的覆盖了用户刚才会走的那条路径？
- 如果正式地址、手机、浏览器缓存、旧进程或慢响应参与其中，我是否已经在对应层面验证过？
- 如果系统背后还有第三方或远端真相源，我现在看到的是推断值，还是已经对账过的事实？

## 6. 一个好的暂停测试

问自己：

- 我现在是否真的需要用户做决策或授权？

如果不需要，就继续。
