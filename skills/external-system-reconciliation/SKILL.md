# 外部系统对账

当系统的关键事实不完全由本地代码决定时，使用这个技能。

适用例子：

- 第三方 API 集成
- 支付、交易、通知、邮件、短信
- 云资源、部署进程、证书、反向代理
- 远端队列、Webhook、异步投影
- 设备状态、权限状态、外部账户状态

## 核心原则

- 本地日志不是最终真相
- 文案不是最终真相
- 缓存不是最终真相
- 用户会对照哪个系统，就必须以那个系统为准

## 第一步：画出真相源矩阵

对每个关键字段回答：

- 这个字段从哪里来
- 最终以谁为准
- 本地是否只是缓存 / 投影 / 推断
- 写操作后多久能稳定对账

常见字段：

- 状态
- 余额 / 数量 / 权限
- 订单 / 支付 / 通知结果
- 资源是否真的创建、替换、删除
- 对用户展示的摘要 / 理由 / 文案

## 第二步：区分四种结果

不要把所有问题都叫失败。至少分清：

1. 未执行
2. 已执行但尚未投影
3. 外部明确拒绝
4. 本地文案 / 映射 / 解释错误

不区分这四类，后续定位和用户沟通都会失真。

## 第三步：做真实对账

每次关键写操作后，至少检查：

- 本地记录了什么
- 外部系统真实状态是什么
- 两者是否一致
- 若不一致，是延迟、权限、参数、映射还是语义错误

如果外部系统有最终回执、状态码、资源查询接口或控制台状态，优先使用它们，而不是只看应用内部日志。

## 第四步：修正文案与验收口径

如果真实状态和用户看到的文案不一致：

- 先修正系统行为
- 再修正文案
- 再补回归验证

不要只改提示词或通知措辞来掩盖状态机问题。

## 什么时候不能声称完成

- 关键字段还没完成真实对账
- 外部系统状态未知，只是本地猜测成功
- 权限不足 / 配置错误 / 拒单原因没查明
- 用户看到的摘要、盈亏、状态、结果与外部真实值冲突

## 输出要求

汇报时明确区分：

- 本地确认的事实
- 外部确认的事实
- 当前推断
- 未确认项

如果存在延迟一致性，也要说清“何时再查、查哪个系统、以哪个字段为准”。
