# 独立验证

当任务不能只靠实现者自己宣布“应该没问题了”时，使用这个技能。

这个技能的职责不是重复实现，也不是顺着实现者的叙事走一遍 happy path，而是用独立验证视角寻找“它为什么可能还不对”。

## 何时使用

适用于：

- 用户已经亲自验收并报过 bug
- 任务是用户可见行为、正式入口、复杂状态机或关键共享路径
- 任务依赖外部真相源、长运行系统或高风险自动化
- 本次改动虽然小，但影响面可能扩散到多个能力点
- 需要阻止实现者自证完成

不适用于：

- 纯闲聊
- 完全无状态、影响面显而易见的琐碎本地改字

## 核心原则

### 1. 默认站在“它可能还不对”的立场

不要帮实现者解释为什么设计思路是合理的。验证者只关心：

- 用户路径是否真的通
- 负路径是否暴露
- 证据是否足以支撑完成声明
- 哪些能力点仍未覆盖

### 2. 不接受实现解释替代证据

验证输入应尽量基于：

- 需求
- 验收矩阵
- 影响面分析
- 回归点清单
- 测试 / 浏览器 / 对账 / 运行时证据

而不是基于“我刚才是这样改的，所以应该可以”。

### 3. 关闭权不在实现者手里

验证问题必须使用以下生命周期：

- `open`
- `accepted-for-fix`
- `fixed-awaiting-reverify`
- `verified`
- `closed`

规则：

- `Builder` 可以创建修复、补证据，并把状态推进到 `fixed-awaiting-reverify`
- `Verifier` 才能把状态推进到 `verified`
- `Closer` 只在阻断项都已 `verified` / `closed` 时才允许最终签收

如果要声称“已完成独立验证”，`Verifier` 必须是不同于 `Builder` 的执行体。不能用同一个执行体的“重新读一遍”替代独立验证。

如果当前环境无法提供不同执行体的 `Verifier`，就只能把任务状态报告为“独立验证条件缺失，暂不能完成”。

### 4. 小改动也要做影响和回归判断

即使改动很小，也必须回答：

- 它改动了哪个能力点
- 可能波及哪些用户路径 / 运维路径
- 哪些回归点必须重验
- 哪些点本轮不重验，以及为什么

“只是小改动”不能成为不做影响分析的理由。

## 工作流

### Stage 1：读取最小必要输入

先拿到：

- 需求或完成定义
- 验收用例 / 验收矩阵
- 本次改动说明
- 影响面与回归点清单
- Builder 已提供的验证证据

如果这些输入缺失，就先把缺口列出来，不要硬做形式化验收。

### Stage 2：建立能力覆盖账本

至少列出：

- 本轮能力点
- 每个能力点对应的验证方式
- 已覆盖 / 未覆盖
- 风险等级

不要只写“测试都过了”。

### Stage 3：优先找反例

独立验证时，优先检查：

- 用户最可能踩到的真实路径
- 共享路径和回归点
- 负路径、恢复路径、边界路径
- 本地推断与外部真相源是否一致
- 实现者最容易因为上下文连续而忽略的路径

### Stage 4：输出结构化问题

每个验证问题至少包含：

- `issue_id`
- 严重级别
- 能力点 / 用户路径
- 复现步骤或证据缺口
- 当前结果
- 期望结果
- 是否阻断完成

不要只写“这里可能有问题”。

### Stage 5：重验与签收

Builder 修完后，状态只能推进到 `fixed-awaiting-reverify`。

Verifier 必须：

- 复用同一路径重验
- 检查是否补了新的回归证据
- 判断问题是否真的关闭

### Stage 6：上下文边界交接

如果上下文接近上限、会话需要切换或验证工作需要交班，必须生成 `continuation packet`，至少包含：

- 当前验证目标
- 已覆盖的能力点
- 未覆盖的高风险能力点
- 当前 issue ledger
- 哪些问题在等待重验
- 下一步唯一优先动作
- 证据路径
- 当前为什么还不能宣称完成

## 未完成判定

出现以下任一情况，就不能允许宣称完成：

- 还有阻断 issue 处于 `open`、`accepted-for-fix` 或 `fixed-awaiting-reverify`
- 只给了解释，没有给证据
- 回归点清单缺失
- 高风险能力点仍未覆盖
- 上一次用户报出的路径还没被同路径重验
- 独立验证本应触发，但 `Verifier` 与 `Builder` 实际上是同一执行体
- 上下文需要切换却没有 `continuation packet`

## 输出要求

最终输出应明确：

- 本轮独立验证覆盖了哪些能力点
- 开出了哪些 issue
- 哪些 issue 已被重验关闭
- 哪些风险仍未关闭，因此不能收尾
