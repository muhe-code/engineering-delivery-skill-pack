# Project Hub

当用户不想显式记住多个 skill，而是希望“引用一个技能就够了”时，使用这个技能。

这个技能本身不是 `spec-to-ship` 和 `done-means-done` 的替代品，而是它们的单入口编排层：

- 负责判断当前任务是否需要文档链、计划链、执行链
- 负责区分“轻量变更协议”和“重型文档链”，避免小改动被仪式化
- 负责判断是否必须先做 harness 设计，而不是先堆业务实现
- 负责判断是否必须引入独立验证者，而不是让实现者自证完成
- 负责决定是否必须联动 `spec-to-ship`
- 负责确保最终一定落到 `done-means-done`
- 负责在上下文逼近上限时强制生成交接包，而不是把“快满了”当作结束理由
- 负责继续路由到前端、验收、状态化产品、外部真相源等专门 skill

## 何时使用

适用于：

- 用户想只引用一个 skill 来推进复杂工程任务
- 用户经常忘记该用 `spec-to-ship` 还是 `done-means-done`
- 任务可能会从澄清、文档、实现、验证一路推进到收尾
- 需要先判断任务复杂度，再决定流程重量

不适用于：

- 用户已经明确指定只用某个更专门的 skill
- 纯闲聊或一次性概念讨论

## 路由原则

### 1. 默认把它当成“单入口”

当用户引用 `$project-hub` 时，先不要急着编码。先判断：

1. 这是不是一个需要文档链和计划链的工程任务
2. 这次是否需要把模糊需求收敛成书面契约
3. 用户是否期待端到端完成，而不只是一个局部回答

只要答案不是明显的“否”，就默认同时联动：

- `spec-to-ship`
- `done-means-done`

但如果任务属于长运行系统，还不能停在这里；必须先判断是否要额外联动 `long-running-app-harness`。

### 2. 何时必须联动 `spec-to-ship`

出现以下任一情况，就必须启用 `spec-to-ship`：

- 需求仍然模糊
- 任务跨多个阶段、文件或子系统
- 需要 requirements / PRD / tech spec / plan / tracker
- 任务存在明显范围漂移风险
- 用户刚改变了产品形态、边界或验收口径

### 3. 何时只需要 `done-means-done`

若任务已经清楚到不值得起完整文档链，但用户仍要求严格闭环，则至少启用 `done-means-done`。

例如：

- 明确的小修复
- 已知范围的小型实现
- 已有计划下的收尾执行
- 明确目标的文档完善或运维动作

### 3.2 轻量变更协议

当改动很小、范围稳定、验收口径清楚时，不要机械套完整重流程，但也不能因为“只是小改动”就跳过工程思考。

此时至少必须显式产出一个轻量变更记录，覆盖：

- 改动点
- 影响模块 / 能力点
- 可能波及的用户路径或运维路径
- 本次回归点清单
- 哪些点判断为无需回归，以及理由

流程可以轻，影响分析和回归思考不能轻。

### 3.5 何时必须先做 Harness 设计

出现以下任一情况，就不要把任务当普通功能开发处理：

- agent / bot / worker / scheduler / cron / daemon
- 长时间运行服务、队列消费、Webhook、多进程运行时
- 需要多轮观测、恢复、降级、回滚
- 是否正确必须依赖运行时 signals、外部回执或长周期观察

此时默认额外联动：

- `long-running-app-harness`

### 3.8 何时必须引入独立验证

出现以下任一情况，不能只让实现者自己验收：

- 用户已经亲自验收并报过 bug
- 任务是用户可见产品行为、关键交互、复杂状态机或正式入口
- 任务依赖外部真相源、长运行系统或高风险自动化
- 本次改动会改变多个模块、多个能力点或高风险共享路径
- 即使是小改动，也可能波及整个工程中的关键回归面

此时默认额外联动：

- `independent-verification`

### 4. 继续路由到专门 skill

根据任务特征，继续联动：

- 有前端页面、布局、视觉和交互质量：`frontend-design`
- 重点是质感、排版、信息层级和去 AI 味：`design-taste-frontend`
- 需要逐条验收用例矩阵：`acceptance-test-design`
- 有聊天、会话、轮询、流式、附件、生命周期：`stateful-product-validation`
- 依赖第三方 API、远端服务、部署状态或其他外部真相源：`external-system-reconciliation`
- 有 agent、bot、后台调度、长期运行服务或需要先设计 harness 的系统：`long-running-app-harness`
- 需要独立验收、反向找错、阻断错误完成声明：`independent-verification`
- 需要按 session / 日期 / 时间范围复盘历史对话，并据此更新技能：`session-to-skill-evolution`
- 验证失败或出现异常行为：`systematic-debugging`
- 宣称完成前：`verification-before-completion`

## 执行要求

### 1. 先定义完成

在动手前，明确：

- 用户最终想拿到什么
- 这次任务是否需要完整文档链
- 哪些验证能支撑最终完成声明
- 当前更像“从规格到交付”，还是“明确任务的强闭环执行”

### 2. 重流程，但不滥流程

不要为了“统一入口”而把所有任务都强行升格成重文档工程。

判断标准：

- 长运行、持续运行、需要运行时可观测与回滚：先走 `long-running-app-harness`
- 复杂、模糊、跨模块、有漂移风险：走 `spec-to-ship + done-means-done`
- 清晰、局部、范围稳定：走 `done-means-done` 的轻量变更协议，并按需要补专门 skill

### 2.2 `update_plan` 粒度规则

不要把会话里的 `update_plan` 当成完整 WBS。

它的职责是展示当前阶段板，而不是穷举所有子任务。默认规则：

- 小任务：通常用 3-4 个高层步骤
- 中等任务：允许 5-8 个步骤，只要每步仍是可理解的阶段
- 大任务：`update_plan` 只保留阶段级步骤；更细的任务拆解下沉到仓库内 `plan` / `tracker` / checklist

如果任务复杂度已经明显超过 4 个自然阶段，不要为了迎合短列表而硬压成 4 步。相反，如果任务只有单点修改，也不要机械凑满 4 步。

### 2.5 Builder / Verifier / Closer 三段式

对非琐碎工程任务，默认把执行分成三个责任面：

- `Builder`：实现、初步自检、文档回写、把修复状态推进到 `fixed-awaiting-reverify`
- `Verifier`：独立找错，不接受“实现解释”替代证据；负责给出 `verified`
- `Closer`：只根据验证状态、证据完整性和未覆盖风险决定是否允许收尾

如果任务已经触发 `independent-verification`，`Verifier` 必须由不同于 `Builder` 的执行体承担。不能把“同一执行体切换视角”视为独立验证。

如果当前环境无法提供独立 `Verifier`，则只能把任务状态报告为“因验证条件缺失而未完成”，不能绕过这条规则直接收尾。

当 `Verifier` 由子代理或独立执行体承担时，默认采用两段式验证提示：

- 第一轮是开放审计：只给需求、验收标准、影响面、证据和必要文件，不给“我怀疑这里有问题”之类的预设结论
- 第二轮才是定向复核：仅在第一轮后，对特定疑点或修复项追加聚焦检查

不要一开始就把验证者提示成“只检查我已经想到的这几个点”，否则容易漏掉未知未知问题。

### 2.6 上下文边界只触发交接

上下文逼近上限时，不允许把“快满了”当作结束理由。

必须先产出 `continuation packet`，至少包含：

- 当前目标
- 已完成
- 未完成
- 未关闭 verifier issue
- 高风险未覆盖能力点
- 下一步唯一优先动作
- 证据或文档路径
- 当前为什么还不能宣称完成

### 3. 最终完成一定由 `done-means-done` 收口

无论前面是否启用了 `spec-to-ship`，最终都必须满足：

- 范围内工作已闭合
- 必要验证已完成
- 阻断性 verifier issue 已经 `verified` 或 `closed`
- 文档与实现没有明显漂移
- 不把剩余必要动作包装成“可选下一步”

## 推荐用法

用户可以只说：

```text
请使用 $project-hub 处理这个任务。
如果需要文档链，就联动 $spec-to-ship；如果任务已经明确，就至少按 $done-means-done 的标准收口。
```

或者更直接：

```text
Use $project-hub as the single entrypoint for this task.
Decide whether this needs $spec-to-ship, but always finish it to $done-means-done quality.
```
