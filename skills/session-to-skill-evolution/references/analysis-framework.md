# 会话到技能的分析框架

## 1. 先分三层证据

### 会话层

- 用户实际说了什么
- 用户反复强调了什么
- 用户在哪些地方明确表达不满、困惑或不被理解

### 执行层

- 哪些 tool call / command / search 失败了
- 哪些验证缺失
- 哪些问题只在真实路径、正式入口或外部真相源上暴露

### 技能层

- 哪条规则本该存在却不存在
- 哪条规则存在但触发条件太弱
- 哪个能力本应拆成独立 skill，却被塞进主 skill

## 2. 用户指导模式

优先提炼这些模式：

- 用户如何定义“完成”
- 用户如何定义“不是我要的”
- 用户如何表达风险边界
- 用户如何指定作用域
- 用户如何要求证据与验证
- 用户如何表达“不要机械执行”

## 3. Codex 失败模式

优先提炼这些模式：

- 字面完成，但意图落空
- 缺逐条验收用例
- 缺真实路径验证
- 缺外部权威真相源对账
- 范围漂移未更新文档
- 技能路由不够明确
- 会话复盘没有沉淀进长期技能

## 4. 从发现到改动的映射

### 适合改主 skill 的情况

- 这是跨多个工程重复出现的模式
- 它改变了完成定义或流程 gate
- 它会影响多个已有子 skill 的触发时机

### 适合改子 skill 的情况

- 问题高度集中在某类场景
- 不适合污染主 skill
- 需要单独的 workflow、reference 或 script

### 适合新增 skill 的情况

- 现有 skill 无法自然容纳
- 问题已经形成可复用的独立任务类型
- 用户未来可以直接点名调用这一能力

## 5. 完成标准

本类任务只有在以下条件都满足时才算完成：

- 证据被抓全
- 模式被提炼清楚
- 作用域被正确解析
- 技能实际被更新
- 更新后的 skill 已验证通过
