# 作用域规则

## 1. 指定根技能时

如果用户指定了根技能，例如：

- `project-hub`
- `spec-to-ship`
- `done-means-done`

则默认策略是：

1. 以该技能为根
2. 解析其在 `SKILL.md` 中显式引用的技能
3. 递归扩展到子技能、孙技能
4. 只在这个闭包内更新

## 2. 未指定根技能时

默认全局范围为两部分：

- `~/.codex/skills`：当前机器的全局技能集合
- 当前工程里的 `skills/`：如果存在，则视为工程技能集合

## 3. 显式引用优先于猜测引用

优先使用这些显式信号建立依赖边：

- 反引号中的技能名，例如 `` `spec-to-ship` ``
- `$skill-name` 形式
- `SKILL.md` 中明确的“额外使用”“联动”“路由到”语句

不要因为主题相似就自动把无显式引用的技能算进闭包。

## 4. 不要误改作用域外技能

如果某个问题确实存在于作用域外技能，但用户本轮限定了范围：

- 在最终汇报中标记它
- 不要擅自扩大修改范围

## 5. 同名技能

如果全局与工程里同时存在同名技能：

- 记录两个实际路径
- 若用户是在当前工程内工作，优先更新工程内版本
- 若当前工程本身就是技能仓库，则应同时保持仓库版与本机安装版一致
