# 从规格到交付

当任务不只是“写代码”，而是“把一个模糊请求以受控流程推进到可验证交付”时，使用这个技能。

这个技能是流程总控，不替代专门技能。它负责决定下一份必须存在的产物、何时允许进入下一阶段，以及何时阻止流程漂移。

如果任务虽然使用了这个技能，但范围仍相对收敛，可以把文档写成简版；但不得省略影响面、回归点和验证责任划分。

## 主要产物

在仓库内创建并维护以下文档：

- `docs/product/01-requirements/<feature>.md`
- `docs/product/02-prd/<feature>.md`
- `docs/engineering/03-tech-spec/<feature>.md`
- `docs/engineering/04-plan/<feature>.md`
- `docs/engineering/05-execution/<feature>-tracker.md`

用 `scripts/init_feature_docs.py` 先脚手架文档，再写内容。

如果该脚本不存在、不可执行或不适配当前仓库结构，不要卡住；应显式记录这一偏差，并按仓库现有文档目录与命名约定手工创建文档。

开始前阅读：

- `references/stage-contracts.md`
- `references/orchestration-rules.md`
- `references/clarification-gates.md`
- `references/agent-roles.md`（涉及子代理时）
- `references/completion-discipline.md`
- `references/rebaseline-rules.md`

## 语言契约

- 默认使用当前用户会话语言编写所有产品/工程文档
- 当仓库已有明确文档主语言时，跟随仓库约定
- 在没有明确反向约束时，优先使用中文
- 代码标识符、路径、命令可保留英文；正文、标题、表格字段、review 结论、tracker 记录必须符合所选文档语言

## 何时使用

适用于：

- 用户给的是模糊功能请求，并期待 Codex 帮他细化
- 任务跨越多个步骤、文件或子系统
- 用户希望在仓库里形成 requirements、PRD、tech spec 和 plan
- 用户期待纪律化执行、review、验证与回顾
- 用户希望在复杂工作中使用多代理协作

不适用于：

- 验收标准显而易见的单文件小改动
- 只做分析、不需要落仓库产物
- 为一个很小的修复硬套完整文档链条会显得仪式化

## 核心约束

流程必须按顺序推进：

1. 澄清并捕获需求
2. 将需求转换为可交付的 PRD
3. 将 PRD 转换为可实施的技术方案
4. 将技术方案拆成优先验证的实现计划
5. 按任务执行，并持续 review、验证、反思与文档回写

不要因为“看起来已经够清楚”就跳阶段。

当 requirements、PRD、tech spec 与现实实现发生漂移时，不要继续埋头编码；先修文档，再继续。

在进入实现前，必须至少完成一次文档链交叉验证。不要满足于“每份文档单看都像完整”；要求是它们组合后也不互相打架，并且已经把关键实现默认值锁住。

如果需求包含用户可见前端界面，不能把 UI 当作“最后顺手美化一下”。界面结构、视觉方向、信息层级、可读性、响应式和浏览器验收都属于书面契约的一部分，必须进入 requirements、PRD、tech spec、plan 与 tracker。

如果交付物会通过正式地址、部署后的静态资源、手机浏览器、PWA、外部服务、网关、反向代理或长轮询/流式链路被使用，也不能把“部署后行为”视为实现后的附带事项。生产路径验证与本地验证同样属于书面契约。

如果任务在推进过程中发生“产品形态”或“系统边界”的变化，也不能把原先文档默认视为仍然有效。例如：

- iOS App 改成 Web
- 同步请求改成异步轮询 / 流式
- 单机改成多服务器
- 现货改成合约
- 本地工具改成正式部署服务

这类变化不是普通实现细节，而是需要重建书面契约的 rebaseline 事件。

## 运行规则

### 0. 前端 UI 路由规则

只要任务中包含以下任一内容，就不能只靠总控 skill 单独推进：

- Web 页面、组件、控制台、仪表盘、表单、聊天界面
- 布局混乱、信息密度失衡、长内容难读、层级不清
- 美化、重排、重构视觉、改交互、响应式适配
- 需要更强的设计感、品牌感、中文阅读友好度或去 AI 味

处理要求：

- 默认额外使用 `frontend-design`
- 如果重点是审美质感、布局品味、信息编排、节奏控制或“不要 generic AI UI”，再额外使用 `design-taste-frontend`
- 若用户是在探索方向、风格或交互模型，先使用 `brainstorming`，再进入本文流程

不要把“前端设计”降格成实现阶段的样式修补。

### 0.5 真实路径与环境路由规则

只要任务中包含以下任一内容，就必须在文档和执行里显式区分“本地/替身验证”和“真实路径验证”：

- 正式地址、部署、证书、反向代理、PWA、手机浏览器
- 聊天、会话、轮询、流式、重连、长任务、后台状态机
- 新建对象后立即使用、恢复旧对象继续使用、图片/附件、持久化/投影/缓存

处理要求：

- requirements / PRD / tech spec / plan 必须明确列出黄金路径矩阵
- 至少包含：
  - 新建对象后第一次使用
  - 已有对象继续使用
  - 至少一条慢响应或恢复路径
  - 若有正式入口，则包含正式入口验证
- mock / fixture / 本地替身如果缺少真实系统中的关键语义，必须在文档中显式写出差异与补充验证

### 0.6 产品形态变更路由规则

只要出现以下任一情况，就必须停止“沿用旧文档继续做”的惯性，先做 rebaseline：

- 用户把交付形态改成另一种产品，例如原生 App 改 Web
- 用户把核心工作流改成另一种交互模式，例如同步回复改异步轮询 / 流式
- 用户把单实例 / 单环境改成多实例 / 多服务器 / 多工作目录
- 用户把核心业务对象或执行方式改成另一类，例如现货改合约、只读改可写、离线改在线

处理要求：

- 先使用 `references/rebaseline-rules.md` 中的规则判断漂移等级
- requirements / PRD / tech spec / plan / tracker 必须回写新基线
- 在新基线完成前，不要继续堆实现细节

### 0.7 状态化产品与外部真相源路由规则

只要任务中包含以下任一内容，就应额外使用对应子 skill：

- 需要把用户旅程、视觉效果、功能结果、异常分支、特殊案例整理成逐条验收用例
  - 额外使用 `acceptance-test-design`
- 聊天、会话、轮询、流式、附件、长任务、后台状态机、乐观更新、分页加载、会话恢复、断线重连
  - 额外使用 `stateful-product-validation`
- 依赖第三方 API、远端服务、数据库投影、支付/交易/通知/云资源、设备状态、部署进程或任何“外部权威真相源”
  - 额外使用 `external-system-reconciliation`

### 0.8 用户心意与隐含目标路由规则

只要出现以下任一信号，就不能只按字面需求推进，必须显式捕获用户真正想达成的结果：

- 用户说你“不通人意”“不懂我真正想要什么”“像机器人”
- 用户的表面要求和真正成功标准明显不是一回事
- 用户表达了收益、风险、体感、面子、节省时间、减少焦虑、降低认知负担等隐含目标
- 用户真正介意的是“不要发生什么”，而不只是“做成什么”

处理要求：

- requirements 必须单独写出：
  - 显性目标
  - 隐含目标
  - 不可接受的失败形态
  - 情绪 / 业务 stakes
  - 用户最终用什么标准判断“你真的懂我”
- PRD 必须写出“用户为何提出这个需求，而不只是他嘴上说了什么”
- tech spec / plan 必须把这些目标转成可验证行为，而不是停留在沟通措辞
- tracker 必须记录本轮是否出现“字面完成但意图落空”的风险，以及如何关闭

### 0.9 长运行应用与 Harness 路由规则

只要任务中包含以下任一内容，就必须把 harness 设计写进书面契约，而不是后补：

- agent / bot / scheduler / cron / worker / daemon
- 长时间运行服务、队列消费、Webhook、多进程运行时
- 需要持续观测、恢复、降级、回滚的系统
- 系统是否正确，必须依赖运行时 signals、browser driving、外部回执或长周期观察

处理要求：

- 额外使用 `long-running-app-harness`
- requirements / PRD / tech spec / plan / tracker 必须明确记录 harness contract，至少包括：
  - repo 内知识入口
  - 一键启动与隔离策略
  - 机械化约束与 CI gate
  - logs / metrics / traces / health / browser 或等价信号面
  - 能力点覆盖账本与 eval gate
  - kill switch、disable path、rollback playbook
- 不要接受“先把功能做出来，再补观测与回滚”

### 0.95 独立验证与交接路由规则

出现以下任一情况，就必须把独立验证写进文档与计划，而不是默认由实现者自己签收：

- 用户已经报过 bug 或明确表达过不信任
- 任务涉及正式入口、复杂状态机、共享高风险路径
- 任务依赖外部真相源或长运行系统
- 即使改动较小，也可能影响多个能力点或关键回归面

处理要求：

- 额外使用 `independent-verification`
- requirements / PRD / tech spec / plan / tracker 至少明确：
  - Builder / Verifier / Closer 的责任分工
  - `Verifier` 必须是不同于 `Builder` 的执行体；若环境做不到，则完成条件仍未满足
  - verifier issue 生命周期
  - 本轮影响面与回归点清单
  - context rollover 时必须生成的 continuation packet

### 1. 先把需求问到可辩护，而不是问到看起来像有文档

如果用户只给一句话，不要假装已经足够。

要做的是：

- 按决策主题分层澄清，而不是一上来发大问卷
- 可以一轮问 2-4 个紧密相关的问题，但不能把复杂需求压缩成一句象征性追问
- 把未决问题标成 `P0 / P1 / P2`
- 在 `P0` 仍然依赖用户决策时，不得进入下一阶段
- 至少抽取并记录以下 5 类信息：
  - 用户嘴上要什么
  - 用户真正想避免什么
  - 用户如何判断这件事“像人”“专业”“靠谱”
  - 用户愿意接受的代价与不愿接受的代价
  - 哪些词是他在借题发挥，哪些是核心目标本身

`P0` 表示阻塞下一阶段的产品缺口。`P1` 表示可以带着显式假设继续，但必须写入文档。`P2` 表示可延后讨论但要记录。

### 2. 不要把方案假设伪装成产品需求

如果某项内容更像实现偏好、技术选型、数据源选择或架构假设，而不是用户明确确认的产品要求：

- 不能直接当作已确定需求写死
- 应写入“当前假设”或“待确认关键决策”
- 如果它会改变用户可观察行为、权限边界、风险边界或验收方式，就必须回问用户

### 3. 可见进度必须是产物，不是自信

有效阶段产物只有：

- 需求文档
- PRD
- 技术方案
- 实现计划
- 带验证证据的执行 tracker

其中 requirements 与 PRD 不仅要能回答“做什么”，还要能回答：

- 为什么这件事此刻值得做
- 用户真实想得到的最终体验 / 结果是什么
- 哪些失败虽然技术上“完成”，但用户仍会判定为失败

### 4. 不允许静默范围漂移

如果实现过程中冒出新需求：

1. 先判断是否属于当前范围
2. 如果属于，更新 requirements、PRD、tech spec、plan
3. 如果不属于，明确记录为延期项

不要把额外工作偷偷塞进代码里却不更新书面契约。

### 5. 验证与反思是两件事

代码改了不等于任务完成。

每个任务完成前都要：

1. 跑验证
2. 检查 done definition 是否真正满足
3. 检查文档是否仍与实现一致
4. 只有都满足后，才能在 tracker 中标记完成

如果启用了独立验证，tracker 还必须记录：

- 验证问题状态
- 重验结果
- 仍未覆盖的高风险能力点
- 为什么本轮还不能或已经可以收尾

### 5.5 文档链交叉验证规则

当产物链已经包含 requirements、PRD、tech spec、plan 或 tracker 中的任意三份及以上时，不能只逐篇自审，必须做一次跨文档审计。

最低检查面：

1. 关键决策是否一致：
   - 状态机
   - 权限与自动执行策略
   - 外部真相源
   - 失败升级规则
   - 范围内 / 范围外边界
2. 实现契约是否一致：
   - 环境变量
   - 命令名
   - tool id
   - job 名称与频率
   - 关键数据字段
3. 是否存在实现级模糊词却没有默认值或参数表，例如：
   - “前 N 个”
   - “更高频”
   - “超预算”
   - “适当”
   - “默认”
4. 是否仍残留未闭合占位词，例如：
   - `TBD`
   - `TODO`
   - `待确认`
   - `未定`
   - `待补`

若发现冲突或模糊项，不得带着它们进入实现计划或编码。

### 6. 验证失败时先调试，不要连猜带堆

如果验证失败：

- 不要连续叠加猜测性修复
- 先使用 `systematic-debugging`
- 如果根因改变了产品决策或技术设计，先回写文档

### 7. 一直做到书面范围真正关闭

如果还有范围内任务、验证缺口、review 问题或必要文档更新，不要停。

只有在以下情况才暂停并找用户：

1. 下一步被真实的产品意图/业务权衡阻塞
2. 当前计划已经做完，再做会超范围
3. 现有验证面无法支撑下一条声明
4. 架构本身出现疑点，需要重新定范围

## 模式选择

开始执行前要选定模式，并写入 execution tracker。

### 串行模式

默认模式：

- 一个主代理
- 任务逐个执行
- 不做并行实现

适用于文件数量不多、耦合紧、能逐步验证的工作。

### 并行模式

仅在复杂工作或用户明确要求多代理时使用。

允许并行的前提：

- 计划已经存在
- 任务边界明确
- 写入所有权清晰
- 每个切片都能独立验证

如果任务文件重叠严重，或正确性依赖强顺序，就退回串行模式。

## 子代理策略

子代理是加速器，不是总控替代品。

允许的角色：

- requirements-reviewer
- prd-reviewer
- spec-reviewer
- implementer
- quality-reviewer

角色契约见 `references/agent-roles.md`。

可用 prompt 模板：

- `requirements-reviewer-prompt.md`
- `prd-reviewer-prompt.md`
- `spec-reviewer-prompt.md`
- `implementer-prompt.md`
- `quality-reviewer-prompt.md`

主控职责：

- 选择模式
- 准备精确、局部的上下文
- 定义写入所有权
- 收集 reviewer 结论
- 判断是否必须重写文档
- 在任何完成声明前亲自运行或确认最终验证

## 工作流

### Phase 0：准备工作区

1. 检查仓库上下文
2. 确定 repo root
3. 用 `scripts/init_feature_docs.py` 初始化文档
4. 记录语言选择与模式选择
5. 如果当前范围包含前端 UI，先显式载入 `frontend-design`；当任务强调审美/布局重构时，再额外载入 `design-taste-frontend`
6. 若准备进入实现，且用户未明确要求在当前分支直接操作，则使用 `using-git-worktrees`

如果 `scripts/init_feature_docs.py` 不存在：

7. 记录为 deviation
8. 按仓库既有目录结构手工创建文档
9. 不得因为脚手架缺失而跳过产物链

### Phase 1：需求文档

目标：生成一份把模糊性暴露出来，而不是藏起来的需求文档。

动作：

1. 探查现有代码、文档、约定与约束
2. 依据 `references/clarification-gates.md` 追问，直到需求文档可辩护
3. 把未决问题按 `P0 / P1 / P2` 分类
4. 编写 `docs/product/01-requirements/<feature>.md`
5. 自审或 reviewer 审核以下问题：
   - 是否仍存在阻塞 PRD 的 `P0`
   - 是否把方案假设写成了需求
   - 是否缺失角色、场景、权限、风险边界、失败策略、验收信号
   - 如果是前端任务，是否已经明确界面目标、阅读负担、信息层级、响应式范围、视觉方向与浏览器验收信号
   - 是否已经列出黄金路径矩阵，而不是只描述笼统功能
   - 是否已经列出最脆弱路径，例如“新建后首次使用”“首次加载详情”“恢复旧对象继续使用”
   - 如果依赖外部系统，是否已经列出“哪个系统才是权威真相源”，以及用户实际会对照的字段
   - 是否已经形成可执行的验收用例矩阵，而不是只有抽象验收口径
   - 如果属于长运行系统，是否已经明确 operator model、runtime signals、disable/rollback 入口与能力点覆盖账本
6. 若存在阻塞项，继续澄清，不得进入 PRD

### Phase 2：PRD

目标：定义从产品与交付角度看，“完成”到底是什么。

动作：

1. 从需求文档派生 PRD
2. 明确用户可观察行为、边界策略、权限策略、失败策略、人工干预策略
3. 编写 `docs/product/02-prd/<feature>.md`
4. 审核 PRD 是否满足：
   - 工程不需要再猜产品意图
   - 验收标准不是单纯“仓库里存在某些文件”
   - 关键边界决策已经落文档
   - 如果有前端 UI，PRD 已经定义信息架构、关键页面状态、空态/错态/加载态、内容密度策略与视觉验收标准
   - 验收标准已经覆盖真实用户路径，而不是只覆盖 happy path
   - 如果用户将通过正式地址或手机使用，PRD 已经要求正式入口验证
   - 如果依赖外部系统，PRD 已经要求关键用户可见字段与外部权威状态对账
   - PRD 已经把验收用例按“用户动作 / 预期视觉与交互 / 预期功能结果 / 特殊案例”具体化
   - 如果属于长运行系统，PRD 已经定义 operator 视角下的启动、观察、降级、禁用、恢复与回滚标准

若不满足，返回修改 requirements 与 PRD，不得进入 tech spec。

### Phase 3：技术方案

目标：写出一份别人不用临场发挥架构也能实现的设计。

动作：

1. 映射系统边界与相关文件
2. 说明设计、取舍、接口、数据流、测试、失败模式、迁移/回滚
3. 编写 `docs/engineering/03-tech-spec/<feature>.md`
4. 审核方案是否达到可实现状态

如果有前端 UI，tech spec 还必须交代：

- 组件层级与页面骨架
- 关键布局策略与响应式断点
- 长内容、工具输出、表单反馈、空态/错态/加载态如何表现
- 是否需要无头浏览器、截图或可视化回归作为验收面

如果任务包含部署、聊天、会话、轮询、流式、附件或恢复逻辑，tech spec 还必须交代：

- 本地验证、mock 验证、正式入口验证分别覆盖什么
- 新建对象首轮路径与已有对象恢复路径如何验证
- 慢响应、处理中、重试、断开、恢复在 UI 与 API 层如何表现
- 静态资源刷新、部署替换、旧进程/旧缓存风险如何避免
- acceptance fixture 是否复用了生产关键语义，若没有，差异是什么

如果任务依赖外部权威系统，tech spec 还必须交代：

- source-of-truth matrix：每个关键字段最终以谁为准
- 本地缓存 / 派生状态 / 对外展示状态之间的关系
- 写操作后的对账时机、重试策略与不一致处理
- 如何区分“未执行”“执行成功但未投影”“外部拒绝”“文案误报”

如果任务属于长运行系统，tech spec 还必须交代：

- harness contract 的具体实现位置
- logs / metrics / traces / health / browser driving / replay fixture 中至少哪些会被建立
- 哪些能力点通过哪些 signals 被观察
- scheduler / worker / webhook / timer 至少哪一条真实运行路径会被验证
- kill switch、disable path、rollback playbook 具体由什么机制承载

无论是否属于前端或外部系统，tech spec 还必须交代：

- 每类验收用例由哪种验证方式覆盖
- 哪些用例适合单元 / 集成 / 浏览器 / 正式入口 / 人工回放
- 哪些特殊案例必须做回归，不能只靠主 happy path 代表

如果技术方案暴露产品缺口，先回写 requirements / PRD。

### Phase 3.5：文档链交叉验证

目标：确认 requirements、PRD、tech spec 已经形成一套可实现的统一契约，而不是三篇各自正确的孤岛文档。

动作：

1. 建立 critical-decision ledger，至少覆盖：
   - 状态名与状态转移
   - 自动执行与人工干预边界
   - 外部真相源
   - 失败升级规则
   - 关键环境变量 / 命令 / tool id / job 名称
2. 逐份核对这些关键决策是否一致
3. grep 并清理未闭合占位词
4. 扫描实现级模糊词，凡是会阻碍编码的，都必须：
   - 写成初始默认值
   - 或写成参数表
   - 或明确为不阻塞当前阶段的延期项
5. 将发现与修复结果写入 tracker

未通过本阶段，不得进入实现计划或编码。

### Phase 4：实现计划

目标：把技术方案拆成小而可测、可完成的任务。

动作：

1. 调用 `writing-plans`
2. 编写 `docs/engineering/04-plan/<feature>.md`
3. 确保每个任务都写清文件、步骤、验证、done definition
4. 初始化 `docs/engineering/05-execution/<feature>-tracker.md`

实现计划还必须显式承接文档链交叉验证的结果：

- 把锁定后的环境变量、命令、tool id、job 名称写进任务
- 把已锁定的默认值和参数表写进任务，而不是重新留给实现者判断
- 若某处仍有可接受的假设，必须在任务里标明验证方式和关闭条件

实现计划必须把验收用例矩阵拆到任务级，而不是停留在文档总表里。

如果计划里有前端任务，每个相关任务都必须写清：

- 负责的页面/组件/样式文件
- 预期的用户可见变化
- 浏览器级验证方式
- 何时需要再次调用 `frontend-design` 或 `design-taste-frontend` 复核

如果任务涉及正式入口、聊天状态机或对象生命周期，每个相关任务还必须写清：

- 覆盖哪一条黄金路径
- 是否覆盖“新建后首次使用”与“已有对象继续使用”
- 是否需要正式地址回归，而不是只跑本地
- fixture / mock 与生产语义不一致时，补充哪条真实验证

如果任务依赖外部权威系统，每个相关任务还必须写清：

- 对账对象与权威来源
- 变更后如何验证“本地展示”和“外部真实状态”一致
- 哪些字段允许短暂延迟，哪些字段不允许误报

所有相关任务还必须写清：

- 关联哪些验收用例编号
- 这些用例分别由谁、用什么方式验证
- 修复 bug 后哪些用例必须做回归

若任务属于长运行系统，实现计划还必须把 harness 任务前置，而不是把它们放到最后：

- 启动与隔离
- 观测面
- 能力点覆盖账本
- eval / replay / 真实触发路径
- kill switch / rollback

没有明确验证的计划，不得执行。

### Phase 5：执行

目标：在不脱离书面契约的前提下完成每个计划任务。

任务循环：

1. 重述任务目标与 done definition
2. 串行实现或派发给边界清晰的 worker
3. 跑该任务验证
4. 如果失败，转入 `systematic-debugging`
5. 如果实现改变了设计或范围，先更新文档
6. 每个有意义的任务或批次后调用 `requesting-code-review`
7. 反思：任务是真的完成，还是只是“表面过绿”
8. 在 tracker 里记录证据与状态

如果用户已经发现 bug，执行循环必须额外包含：

9. 复现用户报告的精确路径，而不是只靠相近路径猜测
10. 修复后把该路径写成可重复的回归验证
11. 若之前的 mock / acceptance 没抓住该问题，记录为何没抓住，并补充验证面

如果当前任务还没有验收用例矩阵或用例编号，则先补齐，再继续实现；不要一边开发一边模糊地“顺手测一下”。

如果任务依赖外部权威系统，执行循环还必须额外包含：

12. 写操作后与权威系统对账，而不是只看本地日志或缓存
13. 如果出现不一致，先分清是投影延迟、权限不足、外部拒绝、字段映射错误还是文案误报
14. 没拿到对账证据前，不要把本地推断写成已完成事实

如果任务触及前端 UI，不要只看测试和接口：

- 至少运行一次浏览器级验证
- 检查界面是否真的改善了层级、节奏、可读性和移动端表现
- 如果用户抱怨的是“杂乱无章”，验证目标应是阅读体验与信息组织，而不是样式文件发生了 diff
- 如果用户将通过正式地址访问，至少运行一次正式地址级验证
- 如果用户将通过手机访问，至少运行一次移动视口或真机等价验证
- 对聊天/会话类产品，不得只验证已有会话；必须验证新建后首轮消息

只要 tracker、plan 或 review 还显示范围内工作未闭合，就继续执行。

当新增或修复行为可以被失败测试表达时，使用 `test-driven-development`。

### Phase 6：诚实收尾

任何完成声明前，都要：

1. 跑完整验证面
2. 对照 requirements、PRD、tech spec、plan 检查最终状态
3. 确认 tracker 中所有条目均已完成或明确延期
4. 调用 `verification-before-completion`
5. 汇报已验证内容、延期项与剩余风险

如果本轮交付物主要是文档链而不是代码，完整验证面至少包括：

- 文档文件是否齐全
- 文档链交叉验证是否通过
- 未闭合占位词是否清零
- tracker 是否记录了 deviation、假设、验证面与剩余风险

其中“完整验证面”在以下场景必须额外包含：

- 有正式地址：正式地址回归
- 有前端 UI：浏览器级或移动视口验收
- 有聊天/会话：新建对象首轮路径 + 已有对象恢复路径
- 有 mock/fixture：至少一条不依赖该 mock 的真实路径验证
- 有外部权威系统：至少一条真实对账验证，确认用户可见状态与权威状态一致

## Review 节奏

- Requirements / PRD：重点看歧义、缺约束、隐藏假设、边界策略
- Tech spec：重点看可实现性、边界、接口、风险
- Plan：重点看任务粒度、验证质量、依赖顺序
- Code：重点看正确性、回归风险、与文档契约的一致性
- Final：重点核对文档与实现是否一致，而不是只看测试绿不绿
- Frontend UI：重点看信息架构、视觉层级、阅读成本、中文排版、空态/错态/加载态、响应式与真实页面观感，而不是只看“功能都在”

## 停止条件

出现以下情况要显式上报：

- 因缺少用户产品决策而无法继续澄清
- 文档来回修三轮仍收敛不了
- 某任务多次验证失败，已经上升为架构问题
- 计划写出的验证无法证明其宣称
- 并行任务不再具备清晰文件所有权

## 完成标准

只有当以下条件都成立时，项目才算完成：

- plan 中所有范围内任务都完成
- 验证证据是新鲜的
- 最终代码与当前文档一致
- 文档漂移已修复
- 延期项被明确列出

只要有一条不成立，就不是完成，只能叫“还在推进”。
