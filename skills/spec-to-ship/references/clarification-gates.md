# 澄清闸门

这份规则定义了需求与 PRD 阶段何时必须继续追问，何时才允许进入下一阶段。

## 一、问题分级

### P0：阻塞项

满足以下任一条件，就属于 `P0`：

- 缺少用户必须拍板的产品决策
- 缺少会改变用户可观察行为的边界条件
- 缺少权限、通知、安全、失败处理等关键策略
- 缺少会影响验收口径的成功定义
- 缺少会让工程方案完全不同的核心约束

只要 `P0` 仍未解决，就不得进入下一阶段。

### P1：重要但可带假设继续

满足以下任一条件，通常属于 `P1`：

- 不会阻塞下一阶段，但会影响实现细节
- 可以先用显式假设占位
- 后续必须回写文档，不允许遗忘

### P2：可延后项

满足以下任一条件，通常属于 `P2`：

- 不影响近期设计与实现
- 属于扩展方向或未来优化
- 只是偏好型决策，不改变当前 MVP 的核心行为

## 二、需求阶段必须覆盖的主题

如果以下主题缺失，应优先检查是否构成 `P0`：

- 用户/角色是谁
- 用户要完成什么核心任务
- 成功结果如何被观察和判断
- 明确不做什么
- 输入数据或外部依赖来自哪里
- 权限与访问边界是什么
- 风险边界、预算边界、操作边界是什么
- 失败时系统如何处理
- 人工是否介入、在什么条件下介入
- 时间或交付约束是什么

## 三、PRD 阶段必须覆盖的主题

如果以下主题缺失，应优先检查是否构成 `P0`：

- 功能需求是否具体到工程不用猜
- 用户可观察验收是否明确
- 边界策略是否明确
- 权限/通知策略是否明确
- 失败与降级策略是否明确
- 人工干预策略是否明确
- MVP 与延期项边界是否明确

## 四、禁止直接脑补的内容

以下内容默认不能直接当成已确认需求：

- 技术选型
- 数据库/缓存/消息队列选择
- 第三方数据源选择
- 算法/模型路线
- 执行频率、阈值、风控参数
- 任何会改变用户行为或验收方式的默认值

除非：

- 用户明确指定
- 仓库已有强约定
- 或该项被清楚标成假设而不是确认需求

## 五、允许进入下一阶段前的最后检查

在推进前，问自己：

1. 还有没有会让用户事后说“这不是我要的”的缺口？
2. 还有没有会让工程被迫替用户做产品决定的缺口？
3. 还有没有关键边界是靠默认脑补，而不是被确认或显式假设？

任一答案为“有”，就继续澄清。
