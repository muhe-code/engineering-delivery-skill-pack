# 完成纪律

`spec-to-ship` 的职责是把工作推到真正终点，而不是只把流程走过一遍。

## 默认继续策略

只要剩余工作已经被以下内容隐含要求，就继续做，不要再征求是否继续：

- 需求文档
- PRD
- 技术方案
- 实现计划
- 未解决的 review 结论
- 缺失的验证证据
- 必须补写的文档回写

不要这样问：

- “要不要我再继续优化？”
- “剩下的 plan items 还要我做吗？”
- “要不要顺手把 reviewer prompt 也接上？”

如果这些就是完成当前范围所必需的，就直接做。

## 什么才算完成

完成意味着：

- 所有范围内计划任务已完成
- 所有必要 review 已处理
- 所有必要验证已运行
- 所有关键验收用例已执行并有结果
- 所有必要文档回写已完成
- 没有把真正属于当前范围的剩余工作伪装成“后续优化”
- 交付物语言符合用户或仓库约定
- 正式入口、真实路径与最脆弱生命周期路径（如果存在）已验证
- 如果依赖外部系统，关键用户可见字段已完成权威对账

## 正当暂停理由

只有以下情况才暂停：

- 的确需要用户做决策
- 继续会超出当前范围
- 破坏性或高风险动作需要明确批准
- 现有验证面不足以支撑下一条声明
- 多次失败表明架构需要重新审视

## 反模式

- 把已暴露的计划步骤称为“未来优化”
- 把 reviewer 未解决问题称为“可选 polish”
- 因为测试通过过一次就宣称完成，而文档仍过期
- 对明显属于当前范围的下一步还去问用户要不要继续
- 文档写成了错误语言，却仍声称任务完成
- 用 mock / fixture / 本地开发验证替代正式入口验证
- 没验证“新建后第一次使用”“恢复旧对象继续使用”就宣称聊天或会话系统完成
- 用户已经报出 bug，却仍用之前的验证结果为完成背书
- 产品形态已经变化，却沿用旧 requirements / PRD / tech spec / plan 继续声称完成
- 本地日志、缓存或文案与外部权威状态不一致，却仍声称系统正确
- 只有抽象“测试通过”结论，没有逐条验收用例结果

## 汇报方式

当范围内还有工作：

- 说清剩什么
- 做完
- 做完后再汇报

当范围内工作全部完成：

- 说明已验证内容
- 说明明确延期项
- 然后结束
