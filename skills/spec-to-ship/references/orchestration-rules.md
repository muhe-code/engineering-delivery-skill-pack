# 编排规则

这些规则用于避免 `spec-to-ship` 变成纯仪式，而是保持真正有约束力。

## 1. 分层澄清，但不要象征性提问

不要一次性丢给用户一整套问卷。

但也不要为了显得“渐进”，只问一句浅层问题就急着写文档。

正确做法：

- 每轮围绕一个决策主题提问
- 每轮可以问 2-4 个强相关问题
- 每轮结束后判断是否清掉了对应主题的 `P0`
- `P0` 未清，就继续澄清，不要跳阶段

## 2. 控制文档漂移

实现可以反过来教会文档新的事实。
禁止的是让文档过期。

如果代码暴露出：

- 缺失的需求
- 变化了的边界条件
- 不同的接口契约
- 新的约束

就先更新对应文档，再继续。

如果变化已经不是“小事实更新”，而是产品形态或系统边界切换，例如原生改 Web、同步改异步、单实例改多实例、现货改合约，则不要只做局部补丁；要按 rebaseline 处理，重写受影响阶段文档与验收矩阵。

## 3. 默认串行，慎用并行

串行是默认，因为更易控制、更易验证。

只有在以下前提下，才应该并行：

- 计划已经清晰切分
- 写入所有权明确
- 集成责任仍由主控承担
- reviewer 能独立评估每个切片

并行是吞吐工具，不是替代规划的借口。

## 4. 在对的阶段 review 对的东西

不要把代码 review 当成发现产品决策缺口的主要场所。

各阶段 review 重点：

- requirements：歧义、隐藏假设、缺角色、缺场景、缺约束、缺验收信号
- PRD：范围边界、用户可观察验收、策略清晰度、权限/通知/失败策略、验收用例是否具体
- tech spec：可行性、边界、接口、风险、用例覆盖映射是否完整
- plan：任务粒度、验证质量、顺序关系、用例编号是否真正落到任务级
- code：正确性、回归风险、是否符合文档契约

## 5. 反思是任务完成检查，不是附加感想

测试通过不等于任务完成。

每次验证后都要问：

- 是否满足了任务写明的 done definition
- 是否还留下了已知缺口
- 是否与文档发生漂移
- 是否引入了新的范围内后续工作

只要任何一项暴露缺口，任务就保持打开状态。

## 6. 阻止无限打磨，也阻止过早收工

这个流程是为了交付，不是为了无限润色。

当所有计划任务都完成且验证通过时，项目才结束；但如果：

- 计划本身是错的
- 文档已经过期
- 仍存在关键缺陷

则不能以“都绿了”为由收工。

## 7. 升级触发条件

以下情况应升级给用户：

- 未解决的业务选择阻塞下一阶段
- 多轮失败后已经上升为架构疑问
- 并行 worker 不再具备清晰边界
- 现有验证无法证明原始声明

## 8. 证据胜过叙述

优先相信这些东西，而不是口头自信：

- 仓库内文档
- 测试与命令输出
- review 结论
- execution tracker 记录

如果证据和叙述冲突，以证据为准。

如果本地叙述、缓存状态、通知文案与外部权威系统冲突，也以权威系统证据为准；然后修正文案、映射或投影逻辑。

如果抽象验收标准和逐条用例执行结果冲突，也以逐条用例结果为准；然后回写 PRD、tech spec、plan 或实现。
