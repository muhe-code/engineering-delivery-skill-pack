# Rebaseline 规则

以下情况不是“边做边微调”，而是需要重建当前阶段基线：

- 原生 App 改 Web / 桌面 / CLI
- 同步请求改异步轮询 / 流式 / 后台任务
- 单实例改多实例 / 多服务器 / 多工作目录
- 只读工具改成可写执行器
- 现货改合约、单市场改多市场、离线改正式部署服务

## 1. 先判断变化等级

- `L1`：措辞或局部约束变化，不影响用户可见行为与验收矩阵
- `L2`：影响用户可见行为、核心路径、权限或部署方式
- `L3`：改变产品形态、系统边界、主要状态机或风险边界

`L2 / L3` 都不能只补代码；至少要回写 requirements、PRD、tech spec、plan 中受影响部分。

## 2. 重建哪些东西

至少检查：

- 需求目标是否仍然成立
- 用户关键路径是否变化
- 验收标准是否仍然可辩护
- 技术方案是否仍与现实实现相符
- 计划任务是否仍覆盖新基线
- tracker 里之前的“已完成”是否已失效

## 3. 什么时候才能继续实现

只有当新的书面契约已经足以约束当前工作时，才能继续编码。

不要这样做：

- 先大改实现，最后再补文档
- 把旧 plan 当作大致参考继续往下做
- 用“只是产品方向微调”淡化一次实际上的形态切换
