# 阶段契约

以下是 `spec-to-ship` 必须经过的文档阶段。

## 1. 需求文档

路径：

- `docs/product/01-requirements/<feature>.md`

目的：

- 把模糊请求变成可分析的工程问题
- 尽早暴露缺失信息
- 锁定范围、非范围、约束、开放问题与风险

最低内容：

- 请求摘要
- 业务/用户目标
- 用户或角色
- 核心场景
- 范围内行为
- 范围外行为
- 关键约束
- 验收信号
- 黄金路径矩阵
- 验收用例矩阵（至少包含用户动作、预期结果、特殊案例）
- 当前假设
- 按 `P0 / P1 / P2` 分类的待确认问题
- 仓库上下文说明

进入 PRD 前必须满足：

- 主要歧义已被暴露且有边界
- 所有 `P0` 要么已被用户确认，要么被明确延期且不会影响 PRD
- 没有把实现方案假设伪装成确认后的产品需求

## 2. PRD

路径：

- `docs/product/02-prd/<feature>.md`

目的：

- 定义交付所遵循的产品契约
- 明确从用户视角看什么叫“完成”

最低内容：

- 问题定义
- 产品目标
- 非目标
- 用户/参与方
- 用户故事
- 功能需求
- 关键边界与策略决策
- 验收标准
- 验收用例矩阵
- 成功指标
- MVP 边界
- 明确延期项

进入技术方案前必须满足：

- 工程不需要再猜产品意图
- 验收标准里至少有一部分是用户可观察行为，而不只是仓库里存在文件
- 权限、通知、失败处理、人工干预等关键边界决策已经写清
- 黄金路径已经覆盖“新建后第一次使用”和“已有对象继续使用”（若适用）
- 如果用户将通过正式地址或手机使用，验收标准里已有正式入口验证要求

## 3. 技术方案

路径：

- `docs/engineering/03-tech-spec/<feature>.md`

目的：

- 把 PRD 转成可实施的工程设计
- 降低编码时的架构临场发挥

最低内容：

- 关联源文档
- 当前系统上下文
- 采用的架构与取舍
- 模块与文件边界
- 数据流与状态流
- API / 接口 / 契约变化
- 数据模型变化
- source-of-truth matrix（若涉及外部系统）
- 失败处理
- 观测性设计（若需要）
- 测试策略
- 用例覆盖映射
- 发布 / 迁移 / 回滚方案
- 验证面
- mock / fixture 与真实环境的差异说明
- 已知风险

进入实现计划前必须满足：

- 另一位工程师无需猜核心设计即可实现
- 测试与验证方式已经清楚到足以拆任务
- 至少一条正式路径或生产路径验证已经在设计里有落点（若适用）

## 4. 实现计划

路径：

- `docs/engineering/04-plan/<feature>.md`

目的：

- 把技术方案拆成带客观检查点的可完成任务

每个任务的最低内容：

- 任务目标
- 将创建或修改的文件
- 依赖或顺序说明
- 逐步实施动作
- 验证命令
- 完成定义
- 验证失败时的回退路径
- 该任务覆盖的黄金路径
- 是否涉及正式入口验证
- 是否涉及外部真相源对账
- 关联的验收用例编号

进入执行前必须满足：

- 每个任务都有客观验证
- 任务足够小，不会隐藏子项目

## 5. 执行追踪

路径：

- `docs/engineering/05-execution/<feature>-tracker.md`

目的：

- 保存执行决策与证据账本
- 跟踪验证、review 与文档回写

必备章节：

- 文档语言与执行模式说明
- 任务状态表
- 用例执行台账
- 验证台账
- 文档回写记录
- review 记录
- 最终完成检查表
