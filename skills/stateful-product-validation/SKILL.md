# 状态化产品验证

当任务包含用户可见状态机，而不是单次静态请求时，使用这个技能。

适用例子：

- 聊天 / 会话产品
- 流式回复、轮询、后台长任务
- 新建对象后立即继续操作
- 附件上传、图片消息、草稿态、overlay
- 输入框禁用 / 清空、按钮 loading、分页加载更多、断线重连

## 核心原则

- 不要把“接口返回 200”当作产品完成
- 不要只验证已有对象，跳过“新建后首次使用”
- 不要只验证最终结果，忽略中间用户可见状态
- 用户体感坏掉，哪怕后端没报错，也算没完成

## 先定义验证矩阵

至少覆盖：

1. 新建对象后第一次使用
2. 已有对象继续使用
3. 慢响应 / 处理中 / 重试 / 恢复路径
4. 真实入口验证
5. 移动端或目标视口验证（若适用）

如果任务是聊天 / 会话，还必须覆盖：

- 输入后是否立即有反馈
- 输入框是否按预期清空或保留
- 当前会话是否保持选中
- 中间过程是否只临时展示，最终历史是否正确落账
- markdown / 时间 / 分组 / 分页 / “加载更多” 是否符合产品预期

## 必查状态

不要只看最终内容。逐项检查：

- loading
- empty
- error
- partial
- success
- stale
- retrying
- resumed

如果某个状态在真实产品里会出现，却没有被验证，说明验收面不完整。

## 执行方式

1. 先列出用户真实会走的黄金路径
2. 再列出最脆弱生命周期路径
3. 用浏览器级或等价用户级方式逐条跑
4. 每条路径记录：
   - 入口条件
   - 用户动作
   - 预期中间状态
   - 预期最终状态
   - 实际结果
5. 如果用户已经报 bug，必须按用户原路径复现，不用相近路径替代

## 什么情况下必须判定为未完成

- 发送成功但输入框状态不对
- 数据最终正确，但中途长时间无反馈
- 新建会话可见，但首轮消息发不出去
- 已有会话能看，继续消息失败
- “加载更多” / 分页 / 时间 / markdown / 分组明显坏掉
- 真实入口坏，本地开发态却是好的
- 手机端体感坏，只在桌面验证过

## 证据要求

至少保留一类：

- 浏览器 smoke
- 无头浏览器脚本
- 截图
- 真实接口回放与用户路径说明

如果任务重点是 UI / 体感 / 会话状态，只有单元测试不足以支撑完成声明。
